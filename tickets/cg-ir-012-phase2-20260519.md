# CG-IR-012 Phase 2: next_token IR Liveness Analysis

## next_token IR Structural Summary
The `next_token` function generates a massive linear sequence (588 blocks, ~4200 pre-opt instructions). It consists of a large lexer state-machine loop containing multiple switch dispatches. Due to Loop Invariant Code Motion (LICM) and Profile-Guided Optimization (PGO) block reordering, several variables defined inside the switch/loop bodies are used in the loop headers or preheaders before their lexical definitions in the emitted linear block order.

## Inverted Intervals Found
While strict inverted intervals where ALL uses precede the def were not explicitly found at the textual PRE-OPT IR level, the optimization passes (`symbolic_cfg`, `LICM`, `PGO-BBR`) reorganize the blocks such that the loop preheader and headers execute lexically before the bodies.
This creates variables that have:
- `first_use` in the loop header (e.g., sequence 10)
- `first_def` in the loop body (e.g., sequence 50)
- `last_use` in the loop body (e.g., sequence 100)

## BUG-3 Fix Coverage Assessment
**Verdict: PARTIAL / MISS**

The BUG-3 fix in `compiler_passes.c` relies on the condition:
`if (def_seq[r] >= 0 && last_use[r] >= 0 && last_use[r] < def_seq[r])`

This condition *only* triggers if the **last** use of a variable appears before its first definition. 
However, in a high-frequency loop like `next_token`, a variable is used in the header (seq 10), defined in the body (seq 50), and then **used again** in the body (seq 100).
In this scenario:
- `def_seq` = 50
- `last_use` = 100
Since `last_use (100) < def_seq (50)` is **FALSE**, the BUG-3 fix is entirely bypassed. 
The linear scanner assigns the variable a live interval of `[50, 100]`. 
This leaves the first use at sequence 10 completely outside the live interval. The linear scanner assumes the variable is dead between 0 and 50, allowing it to assign that register (e.g., a callee-saved register like `rbx`) to another temporary.

## Hypothesis for `rbx=-120` Corruption
Because the interval `[10, 50]` is unprotected, the linear scanner safely assigns `rbx` to another temporary variable in that range, which happens to evaluate to `-120` (e.g., a stack offset or local constant). When the loop back-edge jumps back to the header, `rbx` has been overwritten with `-120`, and the unprotected `first_use` at sequence 10 reads this corrupted value, leading to the massive 135MB `munmap` segmentation fault when used as a pointer/size.

## Proposed Fix Delta
We must explicitly track `first_use` to detect loop-spanning inverted intervals.

```diff
  int def_seq[MAX_INSTRS];
  int last_use[MAX_INSTRS];
+ int first_use[MAX_INSTRS];
  for (int i = 0; i < MAX_INSTRS; i++) {
    def_seq[i] = -1;
    last_use[i] = -1;
+   first_use[i] = -1;
  }
...
  if (ins->src0_reg >= 0) {
+     if (first_use[ins->src0_reg] == -1) first_use[ins->src0_reg] = seq;
      last_use[ins->src0_reg] = seq;
  }
...
- if (def_seq[r] >= 0 && last_use[r] >= 0 && last_use[r] < def_seq[r]) {
-     def_seq[r] = last_use[r];
+ if (def_seq[r] >= 0 && first_use[r] >= 0 && first_use[r] < def_seq[r]) {
+     def_seq[r] = first_use[r];
      last_use[r] = seq > 0 ? seq - 1 : 0;
  }
```
